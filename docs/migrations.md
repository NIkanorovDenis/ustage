# Миграции базы данных Bitrix

Миграции используются для переноса структурных изменений между локальной,
development и production базами одним Git-процессом.

## Создание

```bash
./scripts/create-migration.sh add_catalog_color_property
```

Команда создаст PHP-файл в `www/local/migrations`. Измените его под задачу,
проверьте локально, затем добавьте в commit вместе с кодом.

Каждая миграция должна:

- возвращать callable;
- быть идемпотентной — сначала проверять существование создаваемого объекта;
- выбрасывать исключение при ошибке;
- не содержать пароли, токены, заказы или рабочий контент;
- после выполнения никогда не редактироваться. Исправление оформляется новой миграцией.

## Запуск

GitHub Actions запускает runner автоматически после доставки файлов:

- `develop` — на development;
- `main` — на production.

Таблица `ustage_migrations` хранит имя, checksum и время выполнения. Уже
выполненные файлы пропускаются. Если выполненный файл был изменён, деплой
завершается ошибкой.

Локальный ручной запуск из PHP-контейнера:

```bash
docker compose exec php php /var/www/html/local/migrations/runner.php
```

Если Docker требует повышенных прав, добавьте `sudo` перед `docker compose`.

## Что переносить

Подходят свойства и инфоблоки, пользовательские поля, собственные таблицы,
необходимые коду настройки модулей и почтовые шаблоны.

Не переносятся заказы, пользователи, остатки, рабочий контент, файлы `upload`,
пароли и серверные настройки.
